## 렌더링 방식 관련 요약
``` bash
1. 특정 ui 를 만들 때, 뿐 아니라, CSR, SSR 등 렌더링 방식을 공부하기 전에, '페이지 구성 방식' 을 먼저 고려하기

즉, '해당 ui' 에서, '어디까지를 하나의 page' 로 볼 것 인지를 정하면

'어떤 action 을 새로운 페이지로 처리'하고, 
'어떤 action 을 기존 페이지에서의 추가 요청' 으로 볼지가 나뉜다. 


2. 만들고자하는 프로젝트의 서비스가 무엇이 중요한지에 따라서, 렌더링 방식이 달라질 수 있다. 

- 판단 요소는, 'SEO 중요성 정도' , '사용자 빠른 인터랙션 중요성 정도' '사용자별 다른 내용의 중요성 정도' 등으로 볼 수 있을 것 
- 예를 들어서, 
	- 고객 정보가 많은 페이지여서, SEO 는 크게 중요하지 않고, 사용자 인터랙션이 중요하면 -> CSR 
	- SEO 가 중요하면서, 자주 바뀐다면 -> SSR 
	- SEO 중요하지만, 자주 바뀌지 않는 회사 소개 페이지 같은건 - SSG 
	- SEO 도 중요하면서, 사용자 인터랙션도 중요하면 -> 초기 렌더는 SSR, 추후 요청은 CSR 로 처

- 그리고, 이러한 판단은, '페이지 레벨' 에서도 반영될 수 있을 것 같다. 

```

- 이번에 알게 된 것들을 파편적으로 적으면
	- 한큐에 이해될 수 없다는 것 
	- 흐름도를 바탕으로 이해하는게 제일 오래간다는 것
	
	- '초기 렌더링' 과 '추가 요청에 따른 렌더링' 과정으로 나누어서 생각해보면 도움이 된다는 것 
	- '특정 UI' 를 구현할 때, '어디까지를 하나의 PAGE' 로 볼 것 인지 -> SPA, MPA 의 구분 -> 그에 따라서, '사용자의 A탭 클릭' 을 '새로운 page VS 추가 요청에 따른 렌더링' 처리 로 달라질 수 있다는 것.
	- '특정 UI' 를 구현할 때, 'vanilla JS' 로 구현하는 '구체적인 메서드'를 '그룹화' 해보면, react 가 무슨일 하는데? next.js 는? 이런 추상적인 질문에 답할 수 있다. 
	
	- SSR 은 'web apis 등을 활용하여, 브라우저에서 처리할 수 밖에 없는 처리' 를 제외하고, 최대한 브라우저의 렌더링 과정('브라우저 + js 엔진 협업') 을 '미리' 해두려는 것. 
	- SSR 로 '미리 해두면' -> 브라우저에서는 'HTML 렌더' 와 'hydration(html 에 js 로직을 덧붙이는것)' 을 하게 된다는 것. 
	
	- '탭 UI 클릭' 하면 -> 1) UI 의 변화 2) DATA 의 변화 총 2가지의 변화가 초래되는데 -> CSR 에서는 DATA 를 새롭게 받아와서 대응하고, UI 는 기존의 JS 코드로 대응가능 하다는 것. -> 그래서, AJAX 로 '데이터' 만 넘겨받아도, DATA 뿐 아니라 UI 까지 변화될 수 있다는 것.
